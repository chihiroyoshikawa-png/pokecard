# ポケモンタイプ診断アプリ作って酒論で使ってみた話

こんにちは。NEWhでサービスデザイナーをしている吉川です。

突然ですが、あなたの隣にいる同僚、ポケモンでいうと何タイプだと思いますか？

火？　水？　電気？

……考え始めると止まらなくなりません？

私はなりました。そして気がついたら、AIでポケモン風タイプ診断Webアプリを作っていました。しかも、オリジナルのポケモン風クリーチャー画像つき。トレーディングカード風のデザインで、スクショしたらそのままカードゲームで遊べます。

これは、3/30の社内イベント「酒論」で使うために作った診断アプリの制作記録です。

## 🍺 そもそも「酒論」ってなに

NEWhには「酒論」という社内イベントがあります。メンバーが集まって、お酒を飲みながら語り合う場。

せっかくみんなが集まるなら、ただ飲むだけじゃもったいない。**「盛り上がるコンテンツ」を持ち込みたい。**

欲しかった条件は3つです。

- **その人のことを語る時間が生まれること**（お互いを知るきっかけになる）
- **結果が出た瞬間に「たしかに！」と笑えること**
- **スクリーンショットして残したくなること**

……ポケモンタイプ診断、全部いけるのでは？と思いました。

## 🎮 つくったもの

スマホのブラウザで開いて、その場で使える診断アプリです。

### 入力するもの
- **名前**と**性格・魅力**（必須）
- ソフトスキル、ハードスキル、趣味、エピソード（任意）
- **雰囲気のオノマトペ**——ふわふわ、きらきら、めらめら…など9択から1〜2つ
- **キャラクターイメージ**——ネコ、ドラゴン、オオカミ…など12種の生き物から1〜2つ

### 出てくるもの
AIが6タイプ（🔥火・💧水・⚡電気・🕊️飛行・🌿草・❄️氷）のどれかに判定して、トレーディングカード風のカードを生成します。

- ポケモン図鑑風の紹介文（あの独特な空白区切りの文体で）
- オリジナル技3つ（その人のスキルやエピソードから命名）
- 判定理由（カジュアルな解説）
- **AI生成のクリーチャー画像**（選んだモチーフ×タイプ×雰囲気から、世界にひとつだけの生き物が生まれる）

## 🤖 Claudeと一緒に作る——楽しいけど、ツッコミどころも多い

今回もClaude Code（前回の記事で使ったVSCode上のAI開発ツール）と一緒に作りました。

前回の業務経験把握アプリに続いて2作目です。相変わらずコードは読めません。でも「こういうの作りたい」を伝えれば形にしてくれる。そこは信頼している。

……のですが。

今回は**Claudeに「は？」と言いたくなる場面**がけっこうありました。ここからは、そのリアルな格闘記録です。

### 😤 「ポケモン風に」と言ったのに、全然ポケモンじゃない

DALL-E 3（画像生成AI）に「ポケモン風のキャラクター」と指示して、最初に出てきた画像を見て思いました。

**「……これ、ポケモンじゃなくない？」**

目がキラッキラに光ってるし、なんかネックレスつけてるし、全体的に装飾が派手すぎる。ソシャゲのSSRキャラみたいな見た目で、「ポケットモンスター」感がゼロ。

Claudeに「ポケモンっぽくないんだけど」と伝えたら、Claudeがポケモンのデザイン哲学をWeb検索して勉強してくれました。（まじか、調べてくれるのか）

その結果わかったこと：

- **ポケモンの目はシンプル**——キラキラの星型ハイライトじゃなくて、ドット目や小さい楕円
- **アクセサリーは絶対つけない**——ネックレスもスカーフもダメ。すべてのパーツは生物学的な体の一部
- **あえて完璧にしない**——ポケモンの生みの親・杉森建さんの哲学。カッコよすぎるデザインには、わざとダサい要素を足す
- **シルエットテスト**——真っ黒に塗りつぶしても何かわかるくらいシンプルであること
- **色は2〜3色だけ**——フラットで大胆な配色

これ、デザイナーとしてもめちゃくちゃ面白い話でした。**「可愛くしすぎない」「飾りすぎない」が強度を生む**って、サービスデザインにも通じる考え方ですよね。

プロンプトにこれらのルールを反映したら、だいぶポケモンっぽくなりました。「NO accessories, NO jewelry（アクセサリー禁止、宝石禁止）」って英語で怒鳴るように書いたのがよかったのかもしれない。

### 😵 デプロイが魔境だった

ローカル（自分のPC）では動いていたアプリを、**Vercelというサービスで公開する**工程。ここがまあ、しんどかった。

Claudeは「こうすればデプロイできますよ」と手順を3ステップで教えてくれるんです。簡単そうに見える。

でも、**ターミナルの対話型メニュー**が私には鬼門でした。

たとえば環境変数（APIキーをサーバーに登録する作業）の設定画面。

```
? Add OPENAI_API_KEY to which Environments (select multiple)?
❯◯ Production
 ◯ Preview
 ◯ Development
```

**「……どうやって選ぶの？」**

スペースキーで選択、↓キーで移動、Enterで確定。……普段のアプリのUIとは全然違う操作体系。しかも「sensitive（機密）にしますか？」でYesにしたらDevelopment環境が選べなくなってエラー。なんで？

Claudeに「3つとも聞かれなかった」「どうやって指示する？」と何度も聞き返しました。Claudeは丁寧に教えてくれるんですが、**私が今見ている画面とClaude が想定している画面がズレている**ことが何度もあって。

結局、環境変数の設定だけで15分くらいかかりました。コードを書く時間より長い。

ここで思ったのは、**AIが本当に苦手なのは「ターミナルの対話型UI」の操作ガイド**だということ。コードを書くのはめちゃくちゃ得意。でも「今、画面にこう出てるんだけど、次に何を押せばいい？」という質問には、相手の画面が見えないから精度が落ちる。

### 🦊 「かわいい」と「かっこいい」をどう出し分ける？

キャラクター画像を「かわいい系」にするか「かっこいい系」にするか。最初はトグルスイッチ（ON/OFFのボタン）で切り替えようと思ったんですが、それだと直接的すぎてつまらない。

そこでClaude と相談して、**オノマトペで間接的に聞く**方式にしました。

- 「ふわふわ」「にこにこ」「ぽかぽか」→ ゆるかわ系のデザイン
- 「てきぱき」「きらきら」「わくわく」→ 元気キュート系
- 「きりっ」「しゅっ」「めらめら」→ かっこいい系

入力する側も楽しいし、**その人の雰囲気を言語化するプロセス自体が盛り上がる**。直接「かわいいorかっこいい」と聞くより、こっちのほうが断然いい。

こういう**「体験の質を上げるアイデア」は、まだ人間のほうが得意**だと感じます。Claudeは実装は速いけど、「これだとユーザー体験として面白くないよね」という判断は、やっぱりこっちから出さないと出てこない。

## 🃏 スクショがそのままカードになるUI

結果をただテキストで出すのは味気ない。**スクリーンショットしたらそのままカードゲームのカードに見える**デザインにしたかった。

タイプごとに背景のグラデーションが変わり、枠線の色も変わる。HP、技、図鑑テキスト。全部が1枚のカードに収まっています。

酒論で全員分のカードが揃ったら、その場でカードゲームができる。……という妄想をしながら作りました。

## 💡 AIと作って思ったこと

2作目のAI開発をやってみて、前回よりもはっきり見えてきたことがあります。

**AIが得意なこと**
- コードを書く、直す、リファクタリングする
- 知らないことを調べて学習する（ポケモンのデザイン哲学とか）
- 一瞬で動くプロトタイプを出す

**人間がやるべきこと**
- 「これだと体験として面白くないよね」という判断
- 「ポケモンっぽくない」という違和感に気づくこと
- 「オノマトペで間接的に聞いたほうが楽しい」という企画の勘

つまり、**「何を作るか」と「これ違うな」は人間の仕事**で、**「どう作るか」はAIに任せられる**。

前回の記事で書いた「日本語で伝える力と、使ってみて違和感に気づく力。それだけで十分」という感覚は、2作目でさらに確信に変わりました。

## おわりに

ポケモンタイプ診断、技術的にはGPT-4oとDALL-E 3を組み合わせただけのシンプルなアプリです。

でも、**「あの人のことを語る」時間をつくる仕掛け**として設計したことで、ただの診断ツール以上の体験になったんじゃないかと思っています。

作る過程では、AIに「は？」と思う瞬間もありました。でもそれは、AIが悪いんじゃなくて、**私がやりたいことの解像度を上げるプロセス**だったのかもしれない。「ポケモンっぽくない」と感じた瞬間に、じゃあポケモンらしさって何なのかを調べて言語化する。その繰り返しが、結果的にアプリの質を上げてくれた。

アイデアを言語化して、AIと壁打ちしながら形にしていく。
このプロセス自体が、なんだか「酒論」的だったかもしれません。
